# OCR stack

Этот документ фиксирует задел под OCR для сканов и растровых PDF.

## Что есть сейчас

Сейчас проект в основном опирается на текстовый слой PDF и извлечение таблиц через `PyMuPDF`.

Это быстро и стабильно для:

- born-digital PDF;
- документов с нормальным embedded text layer;
- случаев, где OCR вообще не нужен.

Но для сканов и "картинок в PDF" нужен отдельный OCR-движок.

## Что добавлено в проект

В backend добавлен слой конфигурации OCR:

- `backend/ocr_engines.py`
- `settings.OCR_ENGINE`
- `settings.OCR_DEVICE`
- `settings.OCR_LANGUAGES`
- API: `/api/ocr/capabilities`

Это пока не переписывает текущий pipeline, а дает точку расширения:

- можно выбрать активный OCR-движок через env;
- можно проверять доступные движки;
- можно постепенно переключать scanned-PDF сценарии на более сильный OCR.

## Рекомендуемые варианты

Ниже не абстрактный список, а практический shortlist под документы и формы.

### 1. PaddleOCR

Когда брать:

- если нужен сильный и достаточно прикладной OCR для документов;
- если важны CPU-сценарии;
- если хочется потом идти в table/structure пайплайны того же стека.

Почему это хороший кандидат:

- в документации PaddleOCR 3.x описан OCR pipeline с модулями ориентации, unwarping, detection и recognition;
- по их документации PP-OCRv5 стал default и заявлен как заметно лучше PP-OCRv4;
- есть разделение на `mobile` и `server` модели.

Источник:

- PaddleOCR OCR pipeline docs: https://www.paddleocr.ai/v3.0.0/en/version3.x/pipeline_usage/OCR.html

### 2. Surya

Когда брать:

- если основной фокус на document OCR, layout, reading order и table-aware сценариях;
- если готовы к более тяжелому стеку;
- если в будущем нужен не просто текст, а более "умное" понимание страницы.

Почему это интересно:

- официальный репозиторий заявляет OCR, layout analysis, reading order и table recognition;
- Surya 2 позиционируется как единый VLM для layout + OCR + table recognition;
- в README есть явный упор на document OCR, а не на обычные фото.

Источник:

- Surya GitHub: https://github.com/datalab-to/surya

### 3. docTR

Когда брать:

- если нужен аккуратный Python API для экспериментов;
- если хочется гибко выбирать детектор и распознавание;
- если важна удобная интеграция в свой код.

Почему это полезно:

- у docTR удобный end-to-end predictor;
- в документации заявлены pretrained 2-stage OCR predictors;
- проект прямо позиционируется как OCR for documents и сравнивается с cloud OCR.

Источник:

- docTR docs: https://mindee.github.io/doctr/v0.12.0/index.html
- docTR repo: https://github.com/mindee/doctr

## Что я бы выбрал для этого проекта

Практически я бы шел так:

### Базовый путь

- оставить `PyMuPDF` как fast path для born-digital PDF;
- добавить `PaddleOCR` как первый реальный OCR backend для scanned PDF;
- использовать `Surya` как следующий шаг, если понадобится более сильный layout-aware parsing.

### Почему именно так

- у вас много табличных и форменных документов;
- важны скорость и предсказуемость;
- не хочется сразу тащить самый тяжелый стек в production-путь;
- `PaddleOCR` обычно более понятная первая точка интеграции;
- `Surya` выглядит очень сильным кандидатом на более "умный" второй этап.

## Предлагаемая стратегия подключения

### Stage A

Оставить текущую схему как есть:

- `PyMuPDF` для текста и таблиц там, где текстовый слой уже есть.

### Stage B

Добавить fallback для scanned PDF:

- если у первой страницы нет пригодного text layer;
- если confidence/полнота слишком слабые;
- если документ помечен как scan-only.

Тогда:

- прогонять OCR через выбранный backend;
- сохранять OCR text/result рядом с текущими артефактами документа;
- только затем запускать форменную нормализацию.

### Stage C

Если будет нужно лучшее качество по сложным формам:

- отдельно тестировать `Surya` на реальной выборке ваших PDF;
- сравнить его не "вообще", а по вашим метрикам:
  - точность шапки;
  - точность строк таблицы;
  - устойчивость к плохим сканам;
  - время на страницу;
  - потребление памяти.

## Env-переменные

Примеры:

```env
OCR_ENGINE=pymupdf_text
OCR_DEVICE=auto
OCR_LANGUAGES=ru,en,kk
```

Позже можно будет переключать:

```env
OCR_ENGINE=paddleocr
```

или

```env
OCR_ENGINE=surya
```

## Быстрый вывод

Если нужен один практичный совет без долгих исследований:

- для первого реального OCR-фоллбэка бери `PaddleOCR`;
- для более сильного document AI этапа держи в запасе `Surya`;
- `docTR` оставь как удобную исследовательскую и кастомную опцию.
