# Phase 1: объединение форменных фронтов в один `form-portal`

Этот документ описывает первый практический этап наведения порядка:

- без изменения backend-контрактов;
- без изменения пользовательских сценариев;
- без рискованного рефакторинга backend и контрактов;
- с целью убрать главное дублирование во фронтенде.

## Что объединяем

Изначально в проекте было четыре отдельных фронта:

- `frontend_2-19/`
- `frontend_2-43/`
- `frontend_4-20/`
- `frontend_5-52/`

По факту это один и тот же продукт:

- одна и та же панель чтения PDF;
- один и тот же реестр;
- одна и та же карточка документа;
- один и тот же просмотрщик PDF;
- одна и та же логика загрузки;
- одна и та же навигация по другим формам.

Отличия между ними в основном такие:

- дефолтная форма;
- branding и title;
- порт локальной разработки;
- цветовые переменные;
- ссылки на другие домены.

Это значит, что их нужно рассматривать как один `form-portal` с разными конфигурационными профилями.

## Что не трогаем в Phase 1

Чтобы этап был безопасным, специально не трогаем:

- `backend/`
- API-роуты
- структуру БД
- `frontend_analytics/`
- доменную логику по формам
- скрипты backend-обработки

То есть Phase 1 касается только организации кода форменного фронтенда.

## Цель этапа

Вместо четырех копий одного Vue-приложения получить:

- один общий исходный код UI;
- несколько профилей запуска и сборки;
- предсказуемую структуру для дальнейших изменений.

## Каким должен стать результат

Минимальная целевая структура для этого этапа:

```text
apps/
  form-portal/
    package.json
    vite.config.js
    index.html
    src/
      main.js
      App.vue
      styles.css
      components/
        PdfSearchViewer.vue
      config/
        profiles.js
    .env.2-19
    .env.2-43
    .env.4-20
    .env.5-52
```

Идея простая:

- код один;
- профиль формы выбирается через env;
- разные хосты, порты, title и цвета задаются конфигом;
- все визуальные и поведенческие изменения вносятся в одно место.

## Что уже сейчас общее

Судя по текущему состоянию, можно смело считать общим:

- `src/main.js`
- `src/App.vue`
- `src/components/PdfSearchViewer.vue`
- `package.json`
- большая часть `vite.config.js`

То есть переносить придётся в основном не архитектуру, а просто убрать физическое дублирование.

## Что должно стать параметрами

После объединения различия между формами должны жить не в копиях файлов, а в конфиге.

Минимальный набор параметров:

- `VITE_FORM_ID`
- `VITE_FORM_LABEL`
- `VITE_BRAND_TITLE`
- `VITE_SITE_TITLE`
- `VITE_PORT`
- `VITE_ANALYTICS_URL`
- `VITE_ACCENT`
- `VITE_ACCENT_DARK`
- `VITE_ACTIVE_LINK_COLOR`
- `VITE_HERO_A_START`
- `VITE_HERO_A_END`
- `VITE_HERO_B_START`
- `VITE_HERO_B_END`

Дополнительно можно держать в одном JS-конфиге:

- список ссылок на другие формы;
- дефолтные домены;
- подписи в UI;
- ограничения или особенности конкретной формы.

## Что может остаться в коде как form-specific логика

Сейчас в `App.vue` уже есть ветки поведения под разные формы:

- `2-19`
- `2-43`
- `4-20`
- `5-52`

Это нормально.

На этом этапе их не нужно выносить в отдельные модули любой ценой. Достаточно:

- держать их в одном приложении;
- не копировать файл `App.vue` по четырем папкам;
- по возможности постепенно выносить в `src/features/forms/`.

## Предлагаемая структура внутри `form-portal`

Если делать чуть аккуратнее уже на первом этапе, можно сразу мыслить так:

```text
apps/
  form-portal/
    src/
      app/
      components/
      config/
      features/
        documents/
        upload/
        preview/
        forms/
      styles/
```

Но это не обязательное условие Phase 1.

Если нужно минимизировать риск, сначала достаточно просто сложить текущие общие файлы в один каталог.

## Безопасный порядок работ

### Шаг 1. Создать новый каталог приложения

Создать новый каталог:

- `apps/form-portal/`

Скопировать туда один из текущих фронтов как базу.

Лучше взять:

- `frontend_2-43/`

Причина:

- форма `2-43` сейчас выглядит как наиболее "центральная" для проекта;
- на backend уже есть отдельные отчеты и аналитика вокруг `2-43`.

### Шаг 2. Свести различия к env-профилям

Вынести отличия между фронтами в `.env.*` файлы:

- `.env.2-19`
- `.env.2-43`
- `.env.4-20`
- `.env.5-52`

И запускать одно приложение в разных профилях.

Пример логики:

- профиль `2-19` задает `VITE_FORM_ID=form_2_19`, `VITE_FORM_LABEL=2-19`, `VITE_PORT=5174`
- профиль `2-43` задает `VITE_FORM_ID=form_2_43`, `VITE_FORM_LABEL=2-43`, `VITE_PORT=5175`
- профиль `4-20` задает `VITE_FORM_ID=form_4_20`, `VITE_FORM_LABEL=4-20`, `VITE_PORT=5176`
- профиль `5-52` задает `VITE_FORM_ID=form_5_52`, `VITE_FORM_LABEL=5-52`, `VITE_PORT=5177`

### Шаг 3. Унифицировать запуск

После появления `apps/form-portal` запуск должен опираться на один код.

Желаемый смысл скриптов:

- один базовый `npm run dev`
- дополнительные команды вроде:
  - `npm run dev:2-19`
  - `npm run dev:2-43`
  - `npm run dev:4-20`
  - `npm run dev:5-52`

На Windows-уровне `run_frontends.bat` уже должен запускать не четыре разные папки, а один и тот же проект с четырьмя профилями.

### Шаг 4. Проверить полное совпадение поведения

До удаления старых папок нужно проверить:

- загрузку документов;
- список документов;
- карточку документа;
- просмотр PDF;
- пагинацию таблиц;
- form-specific разворачивание строк;
- ссылки на другие формы;
- переход в аналитику.

Только после этого можно считать объединение успешным.

### Шаг 5. Удалить старые дубли форменных фронтов

После проверки единый `apps/form-portal` становится единственным источником правды.

Главное правило:

новая разработка идет только в `apps/form-portal`.

## Что считать критерием успеха

Phase 1 можно считать завершенным, если:

1. код форменного UI живет в одном каталоге;
2. каждая форма поднимается из одного и того же исходника;
3. различия форм задаются конфигом, а не копиями файлов;
4. `run_frontends.bat` больше не зависит от четырех разных `src/`;
5. любое изменение UI вносится в один код и сразу доступно всем формам.

## Риски и как их снизить

### Риск 1. Случайно потерять form-specific поведение

Снижение риска:

- не переписывать `App.vue` с нуля;
- сначала перенести как есть;
- только потом вычищать.

### Риск 2. Сломать dev-запуск

Снижение риска:

- сохранить текущие порты;
- сохранить env-имена;
- менять `run_frontends.bat` только после локальной проверки нового приложения.

### Риск 3. Начать параллельно рефакторить backend

Снижение риска:

- backend не трогаем вообще;
- Phase 1 должен быть только про структуру форменного фронта.

### Риск 4. Слишком рано удалить старые каталоги

Снижение риска:

- сначала добиться стабильной работы нового `form-portal`;
- удалять дубли только после проверки запуска, маршрутов и сборки.

## Что можно сделать сразу после Phase 1

Когда код фронта станет единым, следующий разумный шаг:

- вынести внутри него повторяющиеся участки в `features` и `config`;
- затем уже заняться разложением `backend/` по зонам ответственности;
- потом вынести runtime-данные и образцы из кодовых каталогов.

## Коротко

Phase 1 не про "большой рефакторинг", а про одну очень конкретную вещь:

убрать четыре физические копии одного форменного фронта и заменить их одним приложением с четырьмя профилями запуска.

Это даст наибольшую пользу при наименьшем риске.
