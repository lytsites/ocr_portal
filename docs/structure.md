# Структура проекта

Этот документ фиксирует, как считать текущую структуру проекта, чтобы дальше наводить порядок без резкого рефакторинга и без риска сломать рабочие сценарии.

## Что происходит сейчас

Сейчас в репозитории смешаны несколько разных уровней:

- приложения, которые запускаются отдельно;
- доменная логика по формам документов;
- общая серверная инфраструктура;
- тестовые и рабочие данные;
- служебные скрипты запуска.

Из-за этого одинаковые или близкие по смыслу вещи лежат рядом, но в разных плоскостях.

Примеры:

- исторически `frontend_2-19`, `frontend_2-43`, `frontend_4-20`, `frontend_5-52` были четырьмя почти одинаковыми фронтами для одной задачи;
- `frontend_analytics` это отдельное приложение, но оно лежит на одном уровне с форменными фронтами;
- `backend` одновременно содержит API, OCR-обработку, работу с БД и runtime-данные;
- `forms` содержит PDF-образцы, но по названию похоже на код форм, а не на входные файлы.

## Как считать текущие папки

До рефакторинга договоримся о таком прочтении:

- `backend/`:
  серверное приложение, API, очередь, OCR-обработка, извлечение таблиц, работа с БД и аналитические запросы.
- `apps/form-portal/`:
  единый фронтенд для чтения форм по URL-маршрутам.
- `frontend_analytics/`:
  отдельное аналитическое приложение.
- `forms/`:
  образцы и входные PDF-файлы для тестов, проверки и ручной работы.
- `backend/data/`:
  runtime-данные приложения.
  Здесь сейчас живут БД, загруженные документы и промежуточные результаты.
- `install_autostart.bat`, `restart_site.bat`, `run_backend.bat`, `run_frontends.bat`, `run_workers.bat`:
  служебные скрипты локального запуска, автозапуска и обслуживания.

## Что уже можно считать опорой

В проекте уже есть хорошая база для нормальной структуры:

- формы централизованы через `backend/form_structures.json`;
- описание форм собрано в `backend/form_specs.py`;
- API работает с формами через единый справочник;
- аналитика уже выделяется как отдельный сценарий, а не как часть каждой формы;
- форменные фронты отличаются в основном конфигом, а не архитектурой.

Это значит, что проект удобнее перестраивать не "по файлам", а "по доменам и приложениям".

## Целевая логическая структура

Ниже не обязательное немедленное дерево папок, а целевая модель, к которой стоит двигаться:

```text
ocr-portal/
  apps/
    api/                 # backend как приложение
    form-portal/         # единый UI для форм 2-19 / 2-43 / 4-20 / 5-52
    analytics/           # отдельный UI аналитики
  domains/
    form_2_19/
    form_2_43/
    form_4_20/
    form_5_52/
  shared/
    backend/
    frontend/
  data/
    documents/
    db/
  samples/
    pdf/
  scripts/
  docs/
```

Смысл разделения:

- `apps/` отвечает на вопрос: что запускаем;
- `domains/` отвечает на вопрос: с какими формами и сценариями работаем;
- `shared/` отвечает на вопрос: что используется повторно;
- `data/` хранит runtime-данные и не смешивается с кодом;
- `samples/` хранит тестовые и демонстрационные входные файлы;
- `scripts/` хранит команды запуска и обслуживания.

## Практическая модель без переноса файлов

Прямо сейчас можно считать проект так:

```text
backend/            => будущее apps/api
apps/form-portal/   => текущее единое приложение форменного фронта
frontend_analytics/ => будущее apps/analytics
forms/              => будущее samples/pdf
backend/data/       => будущее data/
*.bat               => будущее scripts/
```

Это уже дает общее понимание, не ломая ни один импорт и ни один путь.

## Правила, которых стоит придерживаться дальше

- Не создавать новые корневые папки по принципу "под каждую форму свой фронт", если это тот же самый интерфейс.
- Не складывать данные, БД и загруженные документы в кодовые каталоги, если можно вынести их в общий `data/`.
- Не класть тестовые PDF рядом с кодом доменной логики.
- Не смешивать доменную логику формы с аналитикой, если аналитика работает поверх уже извлеченных фактов.
- Всё, что общее для форм, должно жить отдельно от конкретной формы.

## Безопасный план миграции

### Этап 1. Зафиксировать правила

Ничего не переносим.

Делаем только организационные выводы:

- считаем `backend` приложением API;
- считаем `apps/form-portal` единым форменным фронтом;
- считаем `forms` каталогом образцов;
- новые модули раскладываем уже по целевой логике, даже если старые папки пока остаются на месте.

### Этап 2. Убрать главное дублирование во фронтенде

Цель:
держать все формы в одном приложении `form-portal`.

Идея:

- один код Vue-приложения;
- разные `env`/порты/branding/дефолтная форма;
- при необходимости разные домены или Vite entrypoint'ы, но не разные копии `src/`.

Что это даст:

- меньше расхождений между формами;
- меньше риска, что фиксы будут внесены только в один фронт;
- проще поддерживать UI и навигацию.

### Этап 3. Разложить backend по смыслу внутри текущей папки

Даже если папка пока остается `backend/`, внутри стоит мыслить так:

- `backend/api/`:
  HTTP-роуты и схемы ответа.
- `backend/services/`:
  orchestration, pipeline, обработка документа.
- `backend/domains/forms/`:
  логика, специфичная для форм.
- `backend/domains/analytics/`:
  логика аналитики поверх фактов.
- `backend/repositories/`:
  работа с MySQL и таблицами.
- `backend/config/`:
  настройки и окружение.

Это можно делать постепенно, без единовременного большого переноса.

### Этап 4. Вынести runtime-данные из кода

Цель:
перестать воспринимать `backend/data/` как часть приложения.

Желаемое направление:

- БД в `data/db/`;
- документы в `data/documents/`;
- временные результаты там же;
- код не зависит от того, что данные лежат именно внутри `backend/`.

### Этап 5. Вынести образцы и служебные скрипты

- `forms/` переименовать по смыслу в `samples/pdf/` или `fixtures/pdf/`;
- bat-скрипты перенести в `scripts/`.

Это косметика, но она сильно улучшает читаемость корня проекта.

## Что не стоит делать резко

- Не переносить всё за один раз.
- Не менять одновременно структуру фронта, backend и пути к данным.
- Не начинать с переименования папок ради красоты.
- Не тащить аналитику внутрь каждой формы, если она уже живет как отдельный сценарий.

Сначала нужно убрать дублирование и договориться о границах ответственности. Уже потом переносить каталоги физически.

## Приоритеты

Если делать по полезности, а не по красоте, порядок такой:

1. объединить четыре одинаковых фронта форм в один код;
2. разложить backend по зонам ответственности;
3. вынести данные и образцы из кодовых каталогов;
4. только потом физически приводить корень репозитория к целевому дереву.

## Коротко

Сейчас проект лучше воспринимать не как "кучу папок", а как:

- одно API-приложение;
- одно форменное приложение;
- одно аналитическое приложение;
- набор доменов по формам;
- runtime-данные и образцы, которые пока лежат слишком близко к коду.

Именно вокруг этого понимания и надо дальше наводить порядок.
